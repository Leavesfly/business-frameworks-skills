# 业务改善样例 · Business Improvement Worked Examples

> `ch5-business-improvement` 的扩展参考。给出 PDCA / RACI / ECRS / Muda 等常用框架的合格填写示范。

---

## 示例场景：某电商公司"售后退款流程"改善

**背景**：客服团队抱怨退款单要经过 5 个人签字，平均处理 3.2 天，用户差评率 18%。

---

## ㊽ PDCA 循环样例

### P (Plan)
- **目标**：平均退款处理时长从 3.2 天降至 0.5 天
- **假设**：80% 的退款属低风险（< 500 元），可自动化
- **方案**：小额退款系统自动审批 + 大额退款双人会签
- **指标**：TAT（周转时长）、差评率、异常退款率

### D (Do)
- W1-W2：梳理历史 10000 单，得出风险分层规则
- W3：上线小额自动审批（≤ 500 元）
- W4：启动 2 周灰度，仅对 Top 20 客户开放

### C (Check)
- 灰度期 TAT 降到 0.3 天（✓ 达标）
- 异常退款率 0.02%（✓ 可控）
- 差评率降至 5%（✓ 超预期）
- **但**：客服满意度无明显改变（原因：仍需处理大额单）

### A (Act)
- 全量放开小额自动退款
- 新课题：大额单审批路径能否从 5 级缩到 3 级？
- 启动下一轮 PDCA

## 52 RACI 样例（售后退款新流程）

| 事项 \ 角色 | 客服主管 | 财务主管 | 风控主管 | 客服专员 | 财务专员 |
|---|---|---|---|---|---|
| 小额退款审批 | I | I | C | **R/A** | — |
| 大额退款审批（≥5000） | C | **A** | C | R | R |
| 异常退款调查 | I | I | **R/A** | C | C |
| 月度规则评审 | **A** | R | R | C | C |

- **R**esponsible 负责执行
- **A**ccountable 总负责（每行最多一个）
- **C**onsulted 被咨询
- **I**nformed 被告知

## 53 Muda / Mura / Muri 样例

### Muda（7 大浪费 TIMWOOD）

| 类型 | 在退款流程的表现 |
|---|---|
| **T** 运输 Transport | 纸质单据楼上楼下流转 |
| **I** 库存 Inventory | 堆积待审批单据（日均 120 单积压） |
| **M** 动作 Motion | 客服反复切换 5 个系统查信息 |
| **W** 等待 Waiting | 平均等主管签字 1.8 天 |
| **O** 过量生产 Overprod. | 风控定期生成无人看的分析报表 |
| **O** 过度加工 Over-proc. | 小额单还走完整 5 级审批 |
| **D** 缺陷 Defects | 错退款返工率 3% |

### Mura（不均）
- 月初退款量 = 月末 4 倍，客服忙闲不均

### Muri（超负荷）
- 大促后一周客服日均处理 120 单，常规仅 30 单，超载 4 倍

## 54 ECRS 样例

按 **E → C → R → S** 的顺序应用（**先问能不能删**）：

| 顺序 | 动作 | 示例 |
|---|---|---|
| **E**liminate 取消 | 能否完全不做？ | 删除小额单的 3 级以上审批 |
| **C**ombine 合并 | 能否合并步骤？ | 合并风控检查 + 财务核算为一步 |
| **R**earrange 重排 | 能否换顺序？ | 让客服先发退款通知再走财务入账 |
| **S**implify 简化 | 能否更简单？ | 用系统模板替代手写邮件 |

**原则**：ECRS 的顺序不可换，"删除"永远优先于"优化"。

## 55 业务改善提案表样例

```
提案编号：BI-2025-042
提案人：客服部-张丽        日期：2025-12-01

【问题现状】
  小额退款 (<500 元) 占 80%，但走完整 5 级审批，平均 3.2 天
  客服日均积压 40 单，用户差评率 18%

【改善建议】
  引入风控自动规则引擎：
  ≤ 500 元 + 客户等级 ≥ B + 无异常行为 → 系统自动退款
  其余维持现行流程

【预期效果】
  TAT: 3.2 天 → 0.5 天（-84%）
  差评率: 18% → 5%
  客服时长节省: 每日 4 小时

【所需资源】
  技术开发 20 人日 | 风控规则评审 5 人日

【责任人 / Deadline】
  技术：王总监 / 2026-01-31 上线灰度
  业务：李经理 / 2026-02-28 全量
```

## ㊻ KPT 样例（周复盘）

```
Keep（保持）
- 自动退款覆盖率从 0 上线到 60%
- 灰度客户满意度得分 8.9/10

Problem（问题）
- 部分商品分类规则还不清晰（生鲜/虚拟品）
- 月底系统压力大，2 次超时触发人工兜底

Try（尝试）
- 下周把生鲜单独规则模板上线
- 与运维对齐压测，目标 10 倍峰值不降级
```

## ㊼ YWT 样例（季度复盘）

```
Y (Yattakoto / 已做)
- Q4 完成退款流程全自动化
- 差评率从 18% → 5%
- 客服产能释放 25%

W (Wakatta / 领会)
- 改善不是"加流程"，而是"删流程"
- 80/20 律：抓好 80% 的简单问题，20% 的复杂问题反而能获得关注
- Owner + 截止日期比方案本身更重要

T (Tsugi / 下一步)
- Q1 启动大额退款会签路径精简（5→3）
- 开发异常预警看板（风控部主导）
- 复盘纪律：每周 30 分钟 KPT，雷打不动
```

## 51 PERT 样例（项目排期）

```
任务                      前置      工期
A 需求梳理                 -         3 天
B 规则设计                 A         5 天
C 技术开发                 B         10 天
D 风控评审                 B         3 天
E 灰度上线                 C,D       5 天
F 全量放开                 E         2 天

关键路径: A → B → C → E → F = 25 天（非 23 天）
浮动任务: D 有 7 天浮动时间，可缓冲
```

## 常见反模式

| 反模式 | 正确做法 |
|---|---|
| PDCA 只有 PD，没有 CA | 必须留时间验证并固化标准 |
| RACI 每人 R+A 都打勾 | 每行只能有一个 A |
| ECRS 直接上优化 | 必须先问"能否删除" |
| 改善提案没有 Owner/Deadline | 提案 = 决议 = 可执行 |
| 复盘全是问题没有"Keep" | Keep 比 Problem 重要，防止全盘重写 |
