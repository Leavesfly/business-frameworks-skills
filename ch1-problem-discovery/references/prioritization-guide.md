## 一、艾森豪威尔矩阵（Urgent × Important）

### 案例 1：产品经理日常工作

```
                急迫
               ↑
  ┌────────────┬─────────────┐
  │ 重要·急迫   │ 重要·不急迫  │
  │ • 线上bug修复│ • Q3 产品规划 │
  │ • 客户投诉处理│ • 用户调研    │
  │ • 版本发布阻塞│ • 竞品分析    │
  ├────────────┼─────────────┤
  │ 不重要·急迫  │ 不重要·不急迫 │
  │ • 临时会议邀请│ • 刷社交媒体  │
  │ • 同事求助   │ • 整理桌面    │
  │ • 邮件秒回   │ • 闲聊        │
  └────────────┴─────────────┘
               │
              重要 →

行动策略：
- 第一象限：立刻做（今天内完成）
- 第二象限：计划做（排入本周日程，每天预留 2h）
- 第三象限：授权做（委托助理或同事，或批量处理）
- 第四象限：删除不做（直接拒绝或忽略）
```

## 二、决策矩阵（Weighted Scoring）

### 案例 1：技术选型——消息队列选择

**背景**：需要为订单系统选择 MQ，候选方案：Kafka / RabbitMQ / RocketMQ

**步骤 1：确定评估维度及权重**
```
维度          权重    说明
─────────────────────────────
吞吐量        30%    峰值 QPS 要求 10w+
延迟          25%    P99 < 10ms
可靠性        25%    不丢消息、事务支持
运维复杂度    20%    团队熟悉度、监控成本
─────────────────────────────
合计         100%
```

**步骤 2：打分（1-5 分，5 分为最优）**
```
方案      吞吐量(30%)  延迟(25%)  可靠性(25%)  运维(20%)  加权得分
───────────────────────────────────────────────────────
Kafka        5          3          4          3       3.95
RabbitMQ     3          5          4          4       3.95
RocketMQ     4          4          5          3       4.05
```

**计算示例（Kafka）**：
`5×0.3 + 3×0.25 + 4×0.25 + 3×0.2 = 1.5 + 0.75 + 1.0 + 0.6 = 3.85`

**结论**：RocketMQ 略胜，但 Kafka 和 RabbitMQ 差距很小，需结合团队技术栈最终决定。

### 案例 2：供应商选择——云服务提供商

**背景**：选择云服务商，候选：阿里云 / 腾讯云 / 华为云

**步骤 1：确定评估维度及权重**
```
维度          权重    说明
─────────────────────────────
价格          35%    三年总拥有成本 TCO
稳定性        30%    SLA 承诺、历史故障率
技术支持      20%    响应速度、专属客服
生态集成      15%    与现有系统兼容性
─────────────────────────────
合计         100%
```

**步骤 2：打分（1-5 分）**
```
方案      价格(35%)  稳定性(30%)  支持(20%)  生态(15%)  加权得分
────────────────────────────────────────────────────────
阿里云       4          5          4          5       4.35
腾讯云       5          4          3          4       4.15
华为云       4          4          5          3       4.05
```

**权重设定过程说明**：
1. 团队讨论列出所有关注点（价格、性能、安全、支持等 8 项）
2. 每人独立给每项重要性打分（1-10 分）
3. 取平均值后归一化为百分比，确保总和 100%
4. CFO 强调成本控制，故价格权重最高；CTO 强调稳定，故稳定性次之

**结论**：阿里云综合得分最高，且生态集成优势明显（已有阿里系产品），建议选用。

## 三、优先级排序的 5 个常见误区

### ❌ 误区 1：把"别人觉得急"当成"真的急"
**表现**：老板/同事说"这个很急"，你就放下手头工作去处理。  
**对策**：问清楚"为什么急？ deadline 是什么？影响范围多大？"，用事实判断而非情绪判断。

### ❌ 误区 2：只做紧急的事，忽略重要的事
**表现**：整天救火，没时间做规划、学习、关系维护等重要但不紧急的事。  
**对策**：每天预留 20% 时间给第二象限（重要·不急迫），这是长期竞争力的来源。

### ❌ 误区 3：完美主义导致拖延
**表现**：总觉得"还没准备好"，一直推迟重要任务的开始。  
**对策**：采用 MVP 思维，先完成再完美。设定"最小可行动单元"，立即启动。

### ❌ 误区 4：不会说不，任务堆积如山
**表现**：对所有请求都说"好"，结果自己的核心目标无法达成。  
**对策**：学会礼貌拒绝或协商："我现在在忙 X，预计 Y 时间后可以处理你的事，可以吗？"

### ❌ 误区 5：优先级一成不变
**表现**：月初定的优先级，月底还在执行，无视环境变化。  
**对策**：每周回顾一次优先级，根据新信息动态调整。优先级是动态的，不是刻在石头上的。

## 「紧急成瘾」自检表

如果你符合以下 3 条及以上，说明你有"紧急成瘾"倾向：

- [ ] 每天下班感觉忙了一天，但说不出完成了什么重要事情
- [ ] 经常被打断，很难有连续 1 小时不被打扰的时间
- [ ] 手机/IM 消息提示音一响就忍不住查看
- [ ] 总是优先回复邮件/消息，而不是推进核心项目
- [ ] 一周下来，重要但不紧急的事项（如学习、规划）进度为零
- [ ] 习惯用"我很忙"作为口头禅，甚至有点自豪
- [ ] 周末也在处理工作消息，无法真正休息

**改善建议**：
1. 关闭非关键通知，设定固定时间段批量处理消息
2. 每天早晨先做 1 件重要·不急迫的事（吃青蛙）
3. 使用番茄钟，保护深度工作时间
4. 每周日晚上 review 本周时间分配，调整下周优先级
