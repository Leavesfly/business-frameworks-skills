## 一、产品策划书模板（67）

```markdown
# [产品名称] 产品策划书

## 1. 背景（1 段）
<行业趋势 + 内部洞察，为什么现在做这件事>

## 2. 用户与问题（1 页）
- 目标用户：Persona 名字 + 具体特征
- 核心痛点：
  1. ___（高频出现）
  2. ___（高影响）
  3. ___（未被满足）

## 3. 机会与规模
- 市场容量：TAM / SAM / SOM（三层漏斗）
- 竞争空白：竞对做了什么 / 没做什么

## 4. 产品定位（一句话）
> 我们是一个______，为______提供______，不同于______。

## 5. 核心功能（MVP 版）
| 功能 | 用户价值 | 优先级 | 工期 |
|---|---|---|---|
| F1 | ... | P0 | 2w |
| F2 | ... | P0 | 3w |
| F3 | ... | P1 | 2w |

## 6. 商业模式（BMC 摘要）
- VP: ...
- CS: ...
- R$: ...
- C$: ...

## 7. 指标与验证（AARRR / North Star）
- 北极星：______
- 近期目标：获取 ___ 用户，激活率 ___%，留存 ___%

## 8. 路线图（里程碑）
- M1 (T+1m)：MVP 内测
- M2 (T+3m)：灰度放开到 ___ 用户
- M3 (T+6m)：商业化验证

## 9. 资源需求
- 产品 1 / 设计 1 / 前端 2 / 后端 2 / 运营 1
- 预算：___ 万（含市场投放）

## 10. 风险与应对（Top 3）
| 风险 | 发生概率 | 影响 | 应对 |
|---|---|---|---|
| 技术卡点 | 中 | 高 | 提前 POC 验证 |
| 监管 | 低 | 高 | 法务前置评估 |
| 竞对抢跑 | 中 | 中 | 分阶段上线、先发制人 |

## 11. 决策建议
请决策：☐ 批准立项  ☐ 调整后再议  ☐ 暂缓
```

## 三、PREP 结构（69）：1 分钟汇报模板

```
【P - Point / 结论】
  "我建议______。"
  一句话，不超过 20 字。

【R - Reason / 理由】
  "有三个原因：
   第一，______；
   第二，______；
   第三，______。"
  最多 3 条，一句一个。

【E - Example / 例证】
  "比如上周我们测试的 ___ 客户，看到了 ___ 的结果。"
  一个具体案例/数据，切忌堆砌。

【P - Point / 重申】
  "所以我的建议还是______。"
  回到开头那句话，让人记住结论。
```

### PREP 实战样例

> **P**：我建议把小额退款的审批权限下放给客服专员。
> **R**：
> - 一，80% 的退款单小于 500 元，风险可控；
> - 二，现在平均处理 3.2 天，拖伤了客户体验；
> - 三，客服团队有能力接得住，历史错审率低于 0.5%。
>
> **E**：我们上周在 Top 20 客户做了 2 周灰度，退款时长降到 0.3 天，差评率从 18% 降到 5%，无一起错退。
>
> **P**：所以建议下周开始全量下放小额退款审批权。

## 四、TAPS 结构（70）：复杂情况汇报模板

```
【T - Topic / 课题】
  "今天想跟你同步一下 X 项目的进展。"
  一句话告知听者"这是什么话题"。

【A - As is / 现状】
  "截至本周，我们已经做到 ___，但同时出现了 ___ 问题。"
  客观描述，不掺杂判断。

【P - Problem / 问题】
  "问题的本质是 ___，如果不解决，会导致 ___。"
  识别核心矛盾 + 不解决的代价。

【S - Solution / 方案】
  "我的建议是 ___，具体分三步：
   第一，______；
   第二，______；
   第三，______。
   需要你的支持是 ______。"
  明确动作 + 明确请求。
```

### TAPS 实战样例（坏消息汇报）

> **T**：跟您汇报一下 X 项目 Q1 上线的风险。
>
> **A**：目前后端开发完成度 60%，原计划 Q1 末上线。但上周供应商 API 变更，我们发现需要改造认证模块，工作量增加 3 周。
>
> **P**：如果按原计划抢工，可能交付一个有安全隐患的版本；如果延期，会错过竞品窗口期。问题本质是认证模块之前评估不足。
>
> **S**：建议分两步走：
> - Q1 末先上线阉割版（砍掉深度集成，保留核心功能），保住窗口期；
> - Q2 第 3 周补齐完整版，正式推广。
>
> 需要您支持的是：① 同意阉割版 PRD；② 批准追加 2 周开发预算。

## 五、TAPS vs PREP 选择指南

| 场景 | 用哪个 |
|---|---|
| 60 秒电梯汇报 | **PREP** |
| 周报 / 月报要点 | **PREP** |
| 争取资源 | **PREP** |
| 坏消息 / 复杂情况 | **TAPS** |
| 跨部门同步 | **TAPS** |
| 项目复盘对外 | **TAPS** |

## 六、通用汇报清单（任何形式都要具备）

- [ ] 一句话结论（先说结论）
- [ ] 数据 ≥ 1 组（拒绝"我感觉"）
- [ ] 具体案例 ≥ 1 个（从抽象到具象）
- [ ] 清晰请求（要什么？何时？Owner 是谁？）
- [ ] 下一步约定（后续节点/复盘时间）

## 七、常见反模式

| 反模式 | 正确做法 |
|---|---|
| 铺陈太多再说结论 | PREP：先给结论 |
| 只说 Problem 不说 Solution | TAPS 必须四段全 |
| 请求模糊（"希望得到支持"） | 必须具体到"您批准 X + 提供 Y" |
| 理由多于 3 条 | 卡死在 3 条，多了记不住 |
| 汇报讲流水账 | 按结构讲，而不是按时间讲 |
