# 第 7 章 · 传达与共享（Communication & Sharing）

**定位**：对外输出。前六章产出的所有结论，最终都要被"讲清楚"才能推动行动。本章给出结构化的书面与口头沟通模板。

**与其他章的关系**：
- ← 所有章节（任何产出都可能需要对外沟通）
- → 沟通后的反馈可能触发大循环，回到 `ch1-problem-discovery`

---

## 框架清单（67–70）

### 67 产品策划书（Product Proposal）
- **用途**：把一个产品想法/立项写成标准化文档，供内部评审、投资方、跨部门协作使用。
- **推荐结构**：
  ```
  1. 一句话概述（电梯演讲）
  2. 背景与机会
     - 市场现状（可引用 ch2 的 PEST / 五力 / SWOT）
     - 目标用户（可引用 ch2 的 Persona / 同理心地图）
  3. 产品定义
     - 解决什么问题（ch1 课题陈述）
     - 核心功能与差异化（ch3 创意单 / ch4 定位）
  4. 商业模式（ch4 BMC）
  5. 竞争分析（ch2 五力 / 定位图）
  6. 路线图与里程碑（ch4 路线图 + 甘特图）
  7. 关键指标（ch4 KPI 树 / AARRR）
  8. 团队与分工（ch6 组织结构图 / RACI）
  9. 资源与预算
  10. 风险与应对（可引用脚本法）
  11. 一页纸摘要（One-pager）
  ```
- **一页纸摘要模板**：
  ```
  产品名：
  一句话：
  目标用户：
  核心价值：
  商业模式：
  3 个月里程碑：
  所需资源：
  ```
- **英文关键词**：product proposal, PRD, one pager

### 68 活动策划书（Event / Campaign Plan）
- **用途**：把一次营销活动 / 线上发布 / 线下活动写清楚。
- **推荐结构（6W2H + 执行表）**：
  ```
  【目的 Why】这次活动要达成什么（业务目标 / 品牌目标）
  【对象 Whom】目标参与者画像
  【内容 What】主题 / 亮点 / 流程
  【时间 When】周期 / 关键节点
  【地点 Where】线上平台 / 线下场地
  【主办 Who】主办方 / 合作方 / 嘉宾
  【方式 How】传播渠道 / 互动形式 / 视觉风格
  【预算 How much】项目预算分解
  ────────────────────────────
  【KPI】参与人数 / 转化率 / 曝光量 / ROI
  【分工】RACI 矩阵
  【排期】甘特图 / 倒计时清单
  【风险】Plan B 预案
  【复盘】活动后 KPT 时间
  ```
- **倒计时清单模板**：
  ```
  | 距离活动 | 任务 | 负责人 | 状态 |
  |---------|------|--------|------|
  | T-30 天 | ...  |        |      |
  | T-14 天 | ...  |        |      |
  | T-7 天  | ...  |        |      |
  | T-1 天  | ...  |        |      |
  | T+1 天  | 复盘 |        |      |
  ```
- **英文关键词**：event plan, campaign plan, marketing event

### 69 PREP（Point / Reason / Example / Point）
- **用途**：让观点在 1–3 分钟内讲清楚的"金字塔式"表达结构。
- **四段**：
  - **P**oint 结论：先说要点 / 主张
  - **R**eason 理由：为什么（1–3 条）
  - **E**xample 例证：具体数据 / 案例 / 故事
  - **P**oint 重申结论：回扣主张 + 行动号召
- **模板**：
  ```
  Point（1 句）：我主张 ___。
  Reason（3 条以内）：
    1. 因为 ___
    2. 因为 ___
    3. 因为 ___
  Example（1–2 个）：比如 ___
  Point（1 句）：所以 ___，建议我们 ___。
  ```
- **适用场景**：汇报开场、会议发言、一对一沟通、邮件主旨段。
- **英文关键词**：PREP method, PREP framework, concise speaking

### 70 TAPS（Target / Actual / Problem / Solution）
- **用途**：问题汇报 / 建议提案的四段式结构，比 PREP 更适合"出了问题要说明"。
- **四段**：
  - **T**arget 目标：原本的目标 / 应该达到的状态
  - **A**ctual 实际：现在的实际情况 / 数据
  - **P**roblem 问题：目标与实际之间的差距 & 原因
  - **S**olution 方案：建议的解决方案 & 下一步
- **模板**：
  ```
  Target：我们原计划在【时间】达成【指标/状态】。
  Actual：目前的实际情况是【数据/现象】。
  Problem：差距是【数值/描述】，根本原因是【分析】。
  Solution：建议【行动 1】【行动 2】，预计【时间】达成【新目标】。
                 所需【资源/决策】，请求【支持/审批】。
  ```
- **与其他框架的关系**：
  - Target–Actual 部分可直接引用 ch1 的 `As is/To be`
  - Problem 部分可引用 ch1 的 `原因分析` / `逻辑树状图`
  - Solution 部分可引用 ch3/ch4 的候选方案
- **英文关键词**：TAPS model, problem reporting, gap solution

## 常用组合

- **产品立项** → `产品策划书`67（整合 ch1–ch4 的所有产出）
- **营销活动** → `活动策划书`68 + `AIDMA`㊴（ch4）
- **会议发言 / 电梯演讲** → `PREP`69
- **问题汇报 / 向上同步** → `TAPS`70
- **完整提案** → 封面用 `PREP`69 开场 → `TAPS`70 说明问题 → `产品策划书`67 详细展开

## 书面输出排版建议

1. **一页纸原则**：所有正式文档都应有"一页纸摘要"
2. **金字塔结构**：结论先行，论据在后
3. **可视化优先**：能用矩阵/图表就不用纯文字
4. **数据要有对比**：目标值 / 实际值 / 行业均值
5. **行动要有 owner**：每条 next step 必须写清"谁、何时、做什么"

## 输出契约

完成本章后必须产出：
1. **一页纸摘要**（视场景选 `产品策划书`67 或 `活动策划书`68 的简版）
2. **至少一份口头表达稿**（`PREP`69 或 `TAPS`70）
3. **行动清单**：每条含 owner + deadline
4. **反馈回收机制**：明确"谁在什么时候给什么反馈" → 用于触发大循环回到 `ch1-problem-discovery`
